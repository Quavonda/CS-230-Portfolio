# CS 230 Portfolio – Module 8 Journal
Software design document and journal submissions for CS-230 Operating Platforms course.

## Client and Requirements
Working on the software design document for The Gaming Room really helped me understand how everything connects when building an application from start to finish. The client wanted to expand their game, Draw It or Lose It, into a scalable, web-based application that could run across multiple platforms. So the goal wasn’t just to make something that works, but something that performs well for multiple users at the same time and can grow over time.

## What I Did Well
One thing I think I did well was breaking down the system into clear parts and thinking about how each piece would function together. Instead of just focusing on coding, I paid attention to the structure, like how the server would handle requests, how memory would be used, and how storage would support all the image data. That made everything feel more organized and easier to understand.

## Design Process Reflection
Going through the design document actually made coding feel less confusing. It gave me a roadmap before jumping into development. Instead of guessing what to build next, I already had a plan laid out, which saved time and helped avoid mistakes. It also made me realize how important planning is, especially for bigger systems.

## What I Would Improve
If I could go back and improve one part, I would probably go deeper into optimization, especially with memory and storage. Since the game deals with a lot of images, I could have explained more about performance strategies like caching or reducing load times. That would make the design even stronger.)

## Understanding User Needs
When it came to understanding the user’s needs, I focused on performance, accessibility, and ease of use. The game had to run smoothly, support multiple players, and respond quickly. That’s important because if the user experience is slow or confusing, people won’t want to use the application no matter how good the idea is.

## My Approach to Design
My approach to designing the software was to think step-by-step: first understand the problem, then break it into smaller parts, and then figure out how those parts connect. In the future, I’d keep using that same approach, but also spend more time comparing different design options before choosing one. That would help me make even better decisions.

## Final Thoughts
Looking back at this project, it really showed me that good software isn’t just about writing code, it’s about planning, understanding the user, and making smart design choices before anything even gets built.
